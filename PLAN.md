## Approach
1. **Add `hasSeenTutorial` to the Firestore profile schema.** No migration needed — `loadProfile` already returns `null`/partial docs, and `saveProfile` merges. Treat `profile?.hasSeenTutorial !== true` as "show it" for both brand-new and pre-existing accounts (one-time for everyone, not just new signups). Make `profile` reactive React state in `App.jsx` (it is currently a local `const` inside the auth-state-change handler — see `App.jsx:38` — and must become `useState` so the UI can react when the flag changes after a save).
2. **Build `TutorialOverlay.jsx`** — a single scrollable modal/card (per user's locked choice), receiving `open`, `isFirstRun` (boolean — distinguishes onboarding from replay), `onComplete` (persists the flag — only invoked for first-run completion), and `onClose` (closes without persisting — used for Escape/backdrop and for every replay interaction) as props from `App`:
   - `TutorialVideo.jsx`: `<video controls preload="metadata" poster="/tutorial-poster-v1.jpg"><source src="/tutorial-v1.mp4" type="video/mp4" /><track kind="captions" src="/tutorial-captions-v1.vtt" srclang="en" label="English" default /></video>` — note `<source>` before `<track>` (conventional element order; both are valid children of `<video>` but sources-then-tracks reads naturally and matches how browsers/tools typically expect the markup) — embedding the **real, finished video** (optimized copy of the user-provided file — see step 6). All three media filenames are versioned (`-v1`) consistently, matching the cache-header strategy in step 6 — an unversioned name on even one of the three would defeat the immutable-cache approach for that asset. A WebVTT captions track is included (see step 6) — the narrated video has no captions otherwise, which is an accessibility gap for deaf/hard-of-hearing users. Keep a lightweight `onError` fallback ("Playback unavailable — here's how it works:" + the written steps remain visible) for genuine runtime failures.
   - Heading "Welcome to CashFlow Tracker" + numbered 4-step copy (locked text below).
   - **The primary button's behavior is decided by `isFirstRun`, not hard-coded**: when `isFirstRun` is true it reads "Got it, let's go →" and calls `onComplete` (persists `hasSeenTutorial: true`); when false (replay) it reads "Close" and calls `onClose` (never persists — replay must never re-write a flag that's already `true`, which was Codex's correctly-flagged bug in the prior draft). A secondary **"Skip"** text link is shown only when `isFirstRun`, and also calls `onClose` without persisting (so an interrupted/skipped onboarding can resurface next session rather than being silently marked complete).
   - Outside-click-to-dismiss + Escape-to-close always call `onClose` regardless of `isFirstRun` — never silently persist.
   - Modal accessibility: `role="dialog"`, `aria-modal="true"`, `aria-labelledby` pointing at the heading, a focus trap (Tab/Shift+Tab cycle within the dialog — implemented as a small local keydown handler over the dialog's focusable elements; no new dependency needed for 2 buttons + a video + a close affordance), focus moved to the dialog on open and restored to the trigger on close, and `document.body` scroll-locked (and unlocked on every dismissal path) — extending `NotificationsPanel.jsx`'s ref/effect pattern with the focus/inertness handling a blocking modal needs that a popover doesn't.
   - **Inerting the background — NOT `#root`**: `index.html`/`main.jsx` mount everything (including `App`, `Shell`, *and* the overlay itself) under a single `<div id="root">`. Marking `#root` `inert` while the overlay is also a descendant of `#root` would inert the overlay too — disabling its own buttons and video controls. Fix: in `App.jsx`'s authenticated render branch, wrap only the `<Shell>…</Shell>` subtree in a dedicated background wrapper (e.g. `<div className="app-background" {...(tutorialOpen ? { inert: '' } : {})}>`), and render `<TutorialOverlay>` as that wrapper's *sibling*, outside it — so the overlay is never a descendant of the element being inerted. **Use conditional prop spreading, not `inert={tutorialOpen}`**: `inert` is a boolean HTML attribute whose mere *presence* makes an element inert — `inert="false"` is still inert — so React's `inert={false}` (which renders `inert="false"`) would not behave as "not inert." Spreading `{...(tutorialOpen ? { inert: '' } : {})}` ensures the attribute is only present when it should be. Add `aria-hidden={tutorialOpen || undefined}` alongside it as the fallback for the (now small) set of browsers/jsdom versions without `inert` support.
3. **Wire it into `App.jsx`**: `App` owns `profile` (now `useState`, not a local `const`) and a `tutorialOpen` boolean, and computes `isFirstRun = !profile?.hasSeenTutorial`. In the authenticated render branch, restructure the existing `<Shell>…</Shell>` return into `<><div className="app-background" {...inertProps}><Shell>…</Shell></div><TutorialOverlay open={tutorialOpen} isFirstRun={isFirstRun} onComplete={handleCompleteTutorial} onClose={() => setTutorialOpen(false)} /></>` — the overlay is a **sibling** of the (potentially inerted) background wrapper, never its descendant (see step 2's inerting note; this is what makes `{...(tutorialOpen ? { inert: '' } : {})}` on the wrapper safe). `tutorialOpen` initializes to `isFirstRun` once the profile loads.
   - **`handleCompleteTutorial` closes synchronously first, persists second**: it immediately calls `setTutorialOpen(false)` and optimistically updates local `profile` state (`setProfile(p => ({ ...p, hasSeenTutorial: true }))`), *then* fires `saveProfile(uid, { hasSeenTutorial: true })` in a `try/catch` (mirroring the existing `catch (err) { console.error(...) }` convention at `App.jsx:75`). This fixes the sequencing bug Codex flagged: an `await`-before-close would leave the modal open (or the UI in an ambiguous state) on a slow/offline connection. Worst case on a failed write: the flag doesn't persist and the overlay reappears next session — a low-stakes, self-healing failure mode consistent with how the rest of the profile-save code already behaves.
   - **Stale-callback guard — generation token, not uid equality**: the existing `onAuthStateChanged` handler (`App.jsx:27`) is async and can resolve after a newer invocation has already started. A `currentUidRef.current === firebaseUser.uid` check (the originally-planned approach) has a gap: if a user signs out and signs back **into the same account**, the ref is repopulated with the *same* uid, so a still-in-flight callback from the *previous* session would pass the equality check and could overwrite fresh state with stale data. Use a monotonically incrementing `authGenerationRef` instead — increment it at the very top of every `onAuthStateChanged` invocation, capture the new value as `myGeneration`, and guard every `set*` call after each `await` with `if (authGenerationRef.current !== myGeneration) return;`. This correctly distinguishes "this exact invocation is still the latest" regardless of whether the uid repeats. **Scope note**: this guard is being added for the two pieces of state this plan introduces (`profile`, `tutorialOpen`); the same theoretical race already exists, unguarded, around the pre-existing `setAuth`/`setUid`/`setAnthropicApiKey`/`setData`/`setLoading` calls in the same handler — that's pre-existing behavior this plan doesn't change or worsen, and retrofitting a guard onto five more call sites of working production code is a separate hardening pass, not a tutorial feature (see Risks). Also explicitly reset `setProfile(null)` / `setTutorialOpen(false)` in the existing sign-out branch (`App.jsx:29`-`34`) alongside the resets already there.
4. **Add a replay entry point**: a new icon button in `Shell.jsx`'s `header-actions`, between `<ThemeToggle />` and `<NotificationsPanel />`. Because `Shell` cannot own state that its parent `App` needs to read, `App` passes an `onOpenTutorial={() => setTutorialOpen(true)}` callback down through `Shell`'s props to the new button's `onClick`. Since `isFirstRun` is computed from `profile.hasSeenTutorial` (already `true` for anyone who can reach the replay button), the same `<TutorialOverlay>` instance automatically renders in "replay" mode — no separate replay-specific component or persistence path needed.
5. **Add `IconHelp`** to `icons.jsx` — a simple circle-with-question-mark SVG, matching the existing 24-icon stroke/viewBox conventions (don't repurpose `IconSparkle`, which is already the app's "AI" visual signifier and would confuse users into thinking the help button is AI-related).

## Key decisions & tradeoffs
- **Embed the real, finished video (re-encoded for web) vs. ship the raw 16.4 MB Remotion render as-is**: The raw render lacks `+faststart` (so playback can't begin until the whole file downloads), uses full-range `yuvj420p` (compatibility risk on some players), and is larger than it needs to be for a UI-chrome screen-recording. Re-encoding with explicit `libx264`/`yuv420p`/`aac`/`+faststart` trades a few minutes of `ffmpeg` work (verified with `ffprobe`) for materially better compatibility, perceived load time, and bandwidth — the right call for a file fetched by every first-time visitor.
- **Versioned filename (`tutorial-v1.mp4`) + immutable cache headers vs. a stable `/tutorial.mp4` URL**: A stable name is simpler to reference but gives Netlify's CDN no signal to bust a cached copy if the video is ever re-recorded; a versioned name lets us set a year-long immutable cache (faster repeat views) without ever risking staleness — bump the suffix on any future replacement.
- **Reuse the Remotion source's on-screen captions for the WebVTT track vs. transcribe the audio fresh**: The narration and on-screen captions in `Tutorial.jsx` were authored together and the `<Sequence>` components give exact frame-accurate timing — re-deriving a transcript from the rendered audio would be slower and risks drifting from what's already shown on screen. This closes Codex's flagged a11y gap (no captions on a narrated video) using data that already exists.
- **`isFirstRun`-driven button behavior owned by `App`, not a separate replay component**: keeps `TutorialOverlay` a single, simple, reusable component (matches the "single scrollable card" decision) while making the persistence semantics impossible to get backwards — the same bug Codex caught twice (round 1 generally, round 2 specifically pinpointing that the revised draft still let replay call `onComplete`) can't recur because there is exactly one code path that calls `saveProfile`, gated on a prop computed in one place.
- **First-login onboarding overlay vs. a dedicated `/tutorial` route or Settings-page entry**: Overlay-on-load maximizes discovery for new users without requiring them to find a menu. Tradeoff: it's an interruption — mitigated by making it dismissible and by gating it on a persisted flag so it only ever appears once (until replayed deliberately).
- **Header icon button vs. Sidebar entry vs. Settings entry**: `Shell.jsx`'s `header-actions` already hosts global, always-visible utility actions (`ThemeToggle`, `NotificationsPanel`); Help belongs there by convention, not buried in `SettingsPanel` (which is scoped to AI-mode preferences) or `Sidebar` (scoped to navigation + account).
- **Single scrollable card vs. multi-step wizard**: A single card is simpler to build and review, and 4 short steps + one video don't justify wizard-style pagination/state machinery.
- **New `IconHelp` vs. reusing `IconSparkle`**: `IconSparkle` already signifies "AI" elsewhere in the UI (see `SettingsPanel`); reusing it for Help would create a misleading association. A small new icon is cheap and keeps semantics clean.
- **Apply to existing users too, not just new signups**: Treating `hasSeenTutorial !== true` uniformly avoids a second code path ("is this a legacy account?") and ensures nobody — old or new — misses the explainer, at the one-time cost of existing users seeing an overlay on their next login after this ships.

## Risks / open questions
- **Pre-existing stale-callback race in `auth`/`uid`/`anthropicApiKey`/`data`/`loading`**: this plan adds a generation-token guard for the *new* `profile`/`tutorialOpen` state (see step 3), because introducing unguarded async state alongside a known-async handler would be careless. But the exact same theoretical race (a slow `onAuthStateChanged` invocation resolving after a newer one has started) already exists, unguarded, for the five pieces of state the original handler sets — this is pre-existing production behavior, not something this change introduces or worsens. Retrofitting the same generation-token guard onto those five call sites would be a defensible follow-up hardening pass, but it's a refactor of working auth-handling code that's out of scope for "add a tutorial" — flagging it here so it's a deliberate, visible deferral rather than something quietly noticed-and-ignored.
- **Caption text is seeded from on-screen captions, not a verified transcript**: I don't have a narration transcript or a speech-to-text tool available, so the WebVTT cues (especially the newly-authored Outro cue, which has no on-screen `<Caption>` counterpart in the Remotion source to anchor it) are a best-effort seed that **must** be checked against the actual narration audio during step 7's manual playback pass and hand-corrected if it drifts. This is called out as a required sub-step, not an optional nicety — shipping unverified captions on an accessibility feature would be self-defeating.
- **Re-encode quality/size tradeoff**: `-crf 23` is a reasonable default for screen-capture content, but the actual output size/quality should be eyeballed after the fact (play it back, check the resulting file size and `ffprobe` codec report) and the CRF adjusted if it looks soft or is still large — this is a "tune once, verify visually" step, not an exact science up front.
- **Caption timing is sequence-level, not word-level**: the WebVTT cues map to each `<Sequence>`'s full on-screen window (frame-accurate per the Remotion source), matching exactly what a viewer already sees as on-screen captions — but if the narration audio's actual speech timing drifts slightly from the sequence boundaries (e.g., a sentence starts a beat before its caption card animates in), the WebVTT cues will be very close but not necessarily frame-perfect to the spoken word. This is the same granularity as the video's existing visual captions, so it's a wash, not a regression — flagging it as a known limitation rather than a defect.
- `saveProfile` writes to Firestore on completion — if that write fails (offline, permissions), the flag won't persist and the overlay reappears next session. Acceptable for v1 (matches how other profile-flag writes in the app already behave — no special retry logic exists elsewhere either), and the synchronous-close-then-async-save ordering means a failed write never traps the user behind the modal.
- Exact visual styling (colors, spacing, animation, focus-ring treatment) isn't pre-specified beyond "follow existing token-based theme" — left to implementation/Codex review to flag if it clashes with `tokens.css` conventions.
- Committing a multi-megabyte binary to the git repo is a one-way door in terms of repo size (history retains every version pushed). Re-encoding once, carefully, before the first commit — and using a versioned filename so future replacements are additive rather than requiring history rewrites — avoids needing a second "oops, re-optimize and force-push" pass later.

## Out of scope

- Localization/translation of tutorial copy or captions into languages other than English.
- Per-tab contextual tooltips or guided product tours beyond this single onboarding overlay.
- Analytics/tracking of tutorial engagement (e.g., "did the user finish the video").
