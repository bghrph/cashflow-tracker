# Plan: Turn CashFlow Tracker into an installable iOS Progressive Web App (PWA)
_Locked via grill ‚Äî by Claude + user_

## Goal
Make CashFlow Tracker (cashflowtracker.net) installable on iOS via Safari's "Add to Home Screen," so it launches like a native app ‚Äî own icon, full-screen standalone window, instant load, usable on flaky connections ‚Äî while keeping the existing browser experience completely intact. Users can freely move between the Safari tab and the home-screen app, always landing on the same account and the same Firestore data ‚Äî same Google sign-in, no special handoff mechanism, and (where iOS preserves it) the same active session. Same visual theme/design as today; the only goal is installability, speed, and resilience.

## Approach
1. **Web App Manifest**: configure `vite-plugin-pwa`'s manifest-generation options (name, short_name, theme/background colors matching `tokens.css`, icons, `display: "standalone"`, `start_url`) as the **single source of truth** for `manifest.webmanifest` ‚Äî do not also hand-author a static manifest file; the plugin generates and injects it, and maintaining both invites drift and ownership conflicts.
2. **Icon & splash assets**: generate the required iOS icon sizes (incl. `apple-touch-icon`) and splash-screen images from the existing branding/logo; store under `public/`.
3. **iOS meta tags & safe areas**: add `apple-mobile-web-app-capable`, `apple-mobile-web-app-status-bar-style`, `apple-touch-icon` links, and `viewport-fit=cover` to `index.html` (the `<link rel="manifest">` tag is injected automatically by `vite-plugin-pwa` per item 1 ‚Äî don't hand-add a second one); add CSS `env(safe-area-inset-*)` padding (top **and** bottom ‚Äî only the bottom nav currently handles an inset) plus `100dvh` so content clears the notch/Dynamic Island/home indicator in standalone mode.
4. **Service worker (app-shell caching, prompt-to-update, explicit cross-tab coordination)**: integrate `vite-plugin-pwa` into `vite.config.js` to precache built HTML/CSS/JS assets. Use a **prompt-to-update** lifecycle (`registerType: 'prompt'` ‚Äî never silent `skipWaiting`) following Vite PWA's `updateSW()` pattern (vite-pwa-org.netlify.app/guide/prompt-for-update.html): show a small "Update available ‚Äî reload" affordance, and only activate after the user accepts. **Correction from round-3 review: that documented flow only reloads the *accepting* tab** ‚Äî it does not by itself coordinate other open tabs/contexts. So add an explicit `BroadcastChannel` (or `localStorage` storage-event) message broadcast from the accepting tab telling sibling same-origin tabs that an update is ready, prompting them to save state and reload too ‚Äî otherwise those tabs keep running old code referencing assets the new SW has already evicted. **Sequencing matters (round 4 refinement): broadcast to siblings *before* calling `updateSW(true)`/activating the waiting worker**, not after ‚Äî activating first would let the new SW start evicting the old caches while sibling tabs are still mid-reload on the old assets.
5. **Firestore offline persistence (correct modular-SDK API)**: enable offline persistence via the installed Firebase v12 modular SDK using `initializeFirestore(app, { localCache: persistentLocalCache({ tabManager: persistentMultipleTabManager() }) })` in `src/lib/firebase.js` ‚Äî **not** `enableIndexedDbPersistence`/`synchronizeTabs`, which are compat-SDK APIs that don't exist in the modular path this app uses. This is what actually gives cached reads + queued writes that auto-sync when back online.
6. **Offline-aware loading & sync-status UI**: replace the indefinite "Loading‚Ä¶" path with explicit states ‚Äî *cache hit* (instant load from persistence), *cache miss while offline* (clear "you're offline and haven't loaded data on this device yet" message + retry), and *reconnect* (resume normal flow). Add a small, centralized save-status indicator (pending / synced / offline-queued / failed) so queued offline writes are visible rather than silent ‚Äî this also closes the existing gap where `saveFirestoreData` is fire-and-forget with no error surface.
7. **Sign-out hygiene for cached data (best-effort, write-safe, init-time-aware)**: on sign-out, first **freeze further mutations** (block new writes from being queued), then `await Promise.race([waitForPendingWrites(), timeout(N seconds)])` ‚Äî an explicit timeout, since `waitForPendingWrites()` only accounts for writes pending *at the moment it's called* and would otherwise hang indefinitely offline. If the timeout wins while offline, warn the user that signing out now will discard unsynced changes rather than silently destroying them. Only then attempt to clear the local Firestore persistence cache (terminate-and-clear pattern). Treat the clear as **best-effort, not guaranteed**: `clearIndexedDbPersistence` requires every Firestore instance on that IndexedDB to terminate first, so a second open Safari tab/installed-app context will make it throw `failed-precondition` ‚Äî document that each Safari/PWA context logs out (and clears) independently, and force-reload the current context after a successful clear. Also add a visible, documented **persistence preference (read pre-init, not a Setup-only toggle)**: store the user's local-persistence choice (default **on**) in `localStorage` and read it *before* the app's first `initializeFirestore` call, since **Firestore cache settings are immutable once the client initializes** ‚Äî a Setup-tab control that only takes effect post-load can't undo caching that already happened on this session's startup. Flipping the preference off requires the same full terminate‚Üíclear‚Üípersist-preference‚Üíreload sequence as sign-out's cache-clear (not a soft in-place toggle); the Setup tab surfaces this control with a one-line explanation of what's cached locally (financial data and the user's Anthropic API key, per `SetupTab.jsx`) and a note that the change takes effect on next reload.
8. **Route-level code-splitting**: convert the four tab components (`OverviewTab`, `TransactionsTab`, `GoalsTab`, `SetupTab`) to `React.lazy` + `Suspense`, so users only download the tab they're viewing.
9. **Lazy-load the tutorial**: defer `TutorialOverlay`/`TutorialVideo` loading until it's actually opened (first-login or help button), keeping it off the critical initial-paint path.
10. **Fonts**: self-host and subset the Google Fonts currently loaded via CDN (`index.html`) so they can actually be precached by the service worker for offline use; fall back to the system font stack if a font fails to load. ("Precache font assets" only works for assets the build controls ‚Äî CDN fonts aren't one today.)
11. **Netlify cache-header policy (per asset type, not one blanket rule)**: `sw.js`/`index.html` ‚Üí `no-cache` (always revalidate, so updates are discoverable); `manifest.webmanifest` ‚Üí short-lived/revalidate; hashed build assets (`/assets/*.js`, `*.css`) and icons ‚Üí long-lived `immutable`. Mirrors and extends the cache-header work already done for the tutorial video in `netlify.toml`.
12. **Performance & installability pass**: run Lighthouse (Chrome DevTools or `npx lighthouse`) against the production build; iterate until **Performance ‚â• 90**. Verify installability/manifest correctness via Chrome DevTools' **Application panel** (or PWABuilder) ‚Äî Chrome has removed the dedicated "PWA" category from Lighthouse, so a "PWA score" is no longer a real target to chase.
13. **Manual on-device verification (the real proof of offline/session behavior)**: install via Safari ‚Üí "Add to Home Screen" on an iOS device/simulator; confirm standalone launch (no browser chrome, safe areas correct), confirm the app-shell loads instantly offline (airplane mode) **after at least one successful online launch**, confirm Firestore offline reads/queued writes work and sync/show correct status on reconnect, confirm the update-prompt flow on a new deploy, and observe what happens to the login session when launching from the home-screen icon (see the session-sharing note under Key decisions). **Hard release gate**: standalone Google sign-in must actually work before shipping ‚Äî if `signInWithPopup` is blocked in the installed app (a documented mobile-platform issue per Firebase's own guidance), the plan requires a working `signInWithRedirect` fallback (Firebase's recommended approach for exactly this situation) to be implemented and verified; the release does not proceed until standalone sign-in succeeds one way or the other. **`signInWithRedirect` is not a drop-in swap** ‚Äî round 3 correctly flagged that the redirect flow depends on `authDomain`/redirect-URI configuration and, for non-default `authDomain`s on non-Firebase hosting (this app is on Netlify), can require a same-origin auth-helper proxy (Firebase's redirect-best-practices doc). The gate therefore explicitly includes **verifying these infra prerequisites on the real device** ‚Äî confirming the configured `authDomain` and redirect URIs actually complete the round trip back into the standalone app, and standing up a proxy/self-hosted helper (with the SW's navigation handling and Netlify's routing rules adjusted to exclude its paths) only if the default configuration doesn't. This is exactly the kind of platform-specific unknown step 13 exists to surface empirically rather than assume away.
14. **Tests**: add/update unit tests covering manifest presence, service-worker registration wiring, and the Firestore persistence initialization call (mocked, consistent with existing Vitest patterns) ‚Äî these prove the *configuration* is wired correctly. **Additionally cover the new destructive lifecycle paths introduced by this plan's fixes**, all mocked: pending-write-on-logout ordering (sign-out awaits `waitForPendingWrites()` before clearing, and warns instead of discarding when offline with pending writes), cache-clear failure handling (a `failed-precondition` from `clearIndexedDbPersistence` is caught and surfaced, not thrown), and the update-acceptance callback (the prompt-to-update flow only activates/reloads after explicit user acceptance). These tests intentionally do **not** claim to prove real offline behavior works end-to-end; that's what step 13's manual verification is for (the same split used successfully for the tutorial feature, where caption-sync was verified manually, not via automated tests).
15. **Deploy**: standard `git push origin main` ‚Üí Netlify CI flow, with the same single confirmation gate at push time used for the tutorial feature.

## Key decisions & tradeoffs
- **PWA, not a native iOS app** ‚Äî the only viable path with no Mac, no Apple Developer Program membership ($99/yr), and no interest in cloud Mac CI. Costs $0 and needs no App Store review, at the cost of deep native APIs (true background push, native share-sheet integration) that this app doesn't need.
- **"Switching between app and website" = default PWA behavior, same account/data ‚Äî session-persistence is "best effort," not promised.** Codex's round-1 review surfaced a real platform nuance here: iOS's "Add to Home Screen" copies cookies at install time, while Firebase Auth's default persistence lives in localStorage/IndexedDB ‚Äî storage that *may or may not* carry into the standalone web app's storage partition, and this has changed across iOS versions (see WebKit's iOS 17.2 storage-behavior notes). **Revised plan:** don't engineer around this (a first-party cookie-based session would be a major new backend subsystem, wildly disproportionate to "make the site installable"). Instead, build the PWA normally and **observe what actually happens on a real device** (plan step 13). If the session carries over ‚Äî great, zero extra work. If it doesn't, the fallback costs the user one extra tap: **sign in again with the same Google account** in the installed app ‚Äî same account, same Firestore data, no new engineering, and still squarely "switching between app and website" in spirit (the data and identity are unified either way; only the *session token* might not be). **Whichever way the session-carryover question resolves, standalone sign-in actually succeeding is a hard release gate (see Approach step 13)** ‚Äî Codex round 2 correctly pushed back on "observe and see" being too soft, since mobile platforms can outright block `signInWithPopup`; the plan now requires a working `signInWithRedirect` fallback to be in place and verified before shipping if the popup flow fails. No custom handoff/sync/toggle mechanism is built regardless.
- **Offline strategy = Tier 2 (app-shell caching + Firestore's built-in persistence, via the correct modular-SDK API)** ‚Äî chosen over Tier 1 (shell-only, data still needs network) because Firestore offline persistence (`persistentLocalCache` + `persistentMultipleTabManager` ‚Äî the actual v12 modular API; Codex caught that the original draft named the wrong, compat-only API) gives "view/edit your data with no signal, auto-sync later" for a small, well-defined slice of config. Chosen over Tier 3 (custom sync + conflict resolution) because that's disproportionate to a personal finance tracker where users are usually online and data volume is small. Enabling it does require handling the edge cases it introduces honestly (cold-start cache misses, stale cached secrets, write-visibility) ‚Äî see Approach items 6‚Äì7 and Risks.
- **No push notifications** ‚Äî the existing `generateNotifications()` (`src/lib/notifications.js`) already surfaces budget/goal alerts in-app on open. Real Web Push requires VAPID key management, a Firestore subscription-storage scheme, a Netlify function to send pushes, and a service-worker push handler ‚Äî roughly doubling scope for a "same-day checking is fine" use case.
- **Performance/installability target = Lighthouse Performance ‚â• 90 + manifest/installability verified via Chrome DevTools' Application panel (not a "Lighthouse PWA score")** ‚Äî Codex correctly flagged that Chrome has removed the dedicated PWA category from Lighthouse, so the original "PWA checks passing" criterion named a tool feature that no longer exists. The *intent* (installable, correct manifest, working service worker) is unchanged; only the verification tooling is corrected. Reached mainly via route-level code-splitting (the four tabs are already cleanly separated) and deferring the tutorial video, not exotic tuning.
- **No Ollama / local models** ‚Äî Ollama cannot run inside an iOS Safari PWA (needs desktop/server compute), so it has no viable *runtime* role. As a *dev-tooling* choice (routing implementation through `master-brain`/local models), it would add coordination overhead and a new failure surface to an already-proven Claude-implements/Codex-reviews loop, for no clear benefit.
- **Codex's role = read-only adversarial plan reviewer only** ‚Äî standard Act 2 of this workflow (up to 5 rounds critiquing this plan; Claude is final arbiter). Codex never implements or touches files, exactly as it operated for the tutorial feature.

## Risks / open questions
- **Session sharing between Safari and the installed app is not guaranteed on iOS** (see Key decisions) ‚Äî the single biggest unknown in this plan. It can only be resolved by observing real-device behavior during step 13; the plan is written so that either outcome is fine (worst case: one extra "sign in again" tap, no new architecture).
- **Pre-existing data-integrity exposure, made more visible (not created) by this plan**: the app stores all user data in a single Firestore document (`users/{uid}/appdata/main`) and overwrites it wholesale on every save (last-write-wins) ‚Äî see `App.jsx`/`firestore.js`. This already risks clobbering across multiple browser tabs/devices today; a PWA naturally encourages "browser tab + installed app open side by side," making the scenario more likely to occur. **A proper fix (per-entity documents + live `onSnapshot` + UUIDs instead of sequential IDs) is a full data-model rewrite ‚Äî its own project, not a PWA-installability task**, and is therefore out of scope here. As a cheap, in-scope mitigation, add a lightweight **"online awareness" indicator** ‚Äî *not* a conflict-detection system (a one-shot read has no `updateTime` on the public `DocumentSnapshot`, and can't detect a remote version after a local overwrite has already happened, so the originally-proposed "staleness banner" wasn't actually implementable). Instead, keep a live `onSnapshot` listener active (`includeMetadataChanges: true`) and compare a stored `updatedAt` field **plus a per-tab write identifier** (`lastWriterId` ‚Äî a fresh nonce generated **per active runtime/tab instance**, not shared device-wide ‚Äî round 4 caught that a device-wide ID would let sibling tabs/contexts suppress *each other's* genuine remote-change notices, defeating the purpose, since each tab would see the other's writes as "self"): if a snapshot reports a newer `updatedAt` whose `lastWriterId` does *not* match this client's own ID, surface a small "this data was changed elsewhere ‚Äî reload to see the latest" notice. The `lastWriterId` check is essential ‚Äî without it, the server-acknowledgment of the *current* client's own pending write (which also bumps `updatedAt` and flips `hasPendingWrites` from true to false) would trigger a false "changed elsewhere" warning about the user's own change. (Caught in round 3 ‚Äî a subtler version of the same "one-shot snapshot can't distinguish self from remote" problem round 2 first flagged.) This tells users *something* changed remotely while they were online; it's an awareness aid, not a guarantee against clobbering. (Flagged by Codex round 1, corrected per round 2; logged as a follow-up project for the real fix.)
- **Pre-existing 1 MiB single-document ceiling**: same family as above ‚Äî every transaction lives in one document, which Firestore caps at 1 MiB. Not specific to or worsened by the PWA conversion; belongs with the data-model rewrite follow-up, not this plan. Noted for awareness only.
- **iOS Safari PWA platform quirks beyond session storage**: cache/storage eviction after periods of inactivity, no true background sync, a more restricted service-worker lifecycle than desktop browsers. These are Apple platform constraints, not implementation bugs ‚Äî document them as known limitations.
- **Icon/splash asset generation**: producing all required iOS sizes from existing branding needs a concrete sourcing plan (e.g., a generation script/tool from the existing logo) ‚Äî to be worked out during implementation, not left ad hoc.
- **Netlify serving of new static assets**: need to confirm `manifest.webmanifest` and the generated service-worker file get correct MIME types and the differentiated cache headers specified in Approach item 11.

## Out of scope
- Native iOS app via Xcode/Swift and App Store distribution (requires a Mac and a paid Apple Developer Program membership ‚Äî explicitly ruled out by the user).
- Any custom handoff/sync/toggle mechanism between "website mode" and "installed app mode," **and** any first-party/cookie-based server session system to force session-sharing on iOS (the plan accepts "sign in again with the same account" as a fine fallback if the platform doesn't share sessions ‚Äî see Key decisions).
- Push notifications / Web Push infrastructure.
- A full custom offline-first sync layer with conflict resolution (Tier 3 offline strategy).
- **The pre-existing single-document/last-write-wins data-model rewrite** (per-entity documents, live `onSnapshot`, UUID-based IDs, collection migration past the 1 MiB ceiling) ‚Äî real and worth doing, but it's a general data-integrity project that exists independently of "make the site installable," not a consequence of this plan. Tracked as a follow-up; this plan only adds a lightweight staleness-warning mitigation.
- Standing up a Firebase-emulator-based integration test suite ‚Äî manual on-device verification (step 13) is this plan's proof of real offline/session behavior, matching the verification approach already used successfully for the tutorial feature.
- Routing any implementation work through Ollama or other local models.
