Data Layer — Firestore Schema

How your data is structured in the cloud, and how the API key is stored

Firestore Collection Structure
📁 users/
└── {uid}  ← one per Google account
├── profile (document)
├── email: "user@gmail.com"
├── displayName: "Hussein"
├── createdAt: timestamp
└── anthropicApiKey: "sk-ant-..." ← user's own key, optional
└── appdata (document — mirrors current localStorage)
├── transactions: [ ...array of tx objects ]
├── budget: { monthly: 3000, rollover: true }
├── recurring: [ ...recurring templates ]
├── goals: [ ...savings goals ]
├── currency: "USD"
├── aiPreference: "fallback"
├── categoryGroups: [ ...groups ]
├── savingsTarget: 500
└── updatedAt: timestamp
🔑 Anthropic API Key Storage

Stored as a field in the user's profile document. Google encrypts all Firestore data at rest. Security rules ensure only the authenticated user can read their own profile — no one else can see it.

On each AI Parse request, the React app reads the key from Firestore (cached in memory after first load) and sends it in the X-Api-Key header to the Netlify Function.

🔒 Security Rules (5 lines)
match /users/{uid}/{doc=**} {
  allow read, write:
    if request.auth != null
    && request.auth.uid == uid;
}

Each user can only touch their own documents. No user can see or modify another user's data.

🔄 Migration: localStorage → Firestore

On first login, the app checks if the user has an existing appdata document in Firestore. If not (new user), it checks if there's data in mbr-data-v4 in localStorage. If found, it offers a one-time "Import your existing data?" prompt — clicking yes uploads it to Firestore and clears localStorage. Existing data is never lost.

💡 Why one document per user (not subcollections)?

The entire app state is read and written together — just like localStorage. A single appdata document keeps the data model identical to the current app, minimising code changes. Firestore's 1MB document limit is generous — a user would need ~5,000+ detailed transactions to approach it.
