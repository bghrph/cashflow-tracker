## Round 2 — Codex
**Material findings**
1. Replay-flag bug: as written, replaying the tutorial via the header button could re-trigger the "mark complete" write path, conflating "user explicitly replayed it" with "first-run completion" and re-persisting `hasSeenTutorial` unnecessarily / resetting other state. Fix: the parent must distinguish *why* the overlay is open (first-run vs. replay) and gate which handler (`onComplete` vs. `onClose`) is wired to which dismissal action accordingly — not leave it to the overlay to infer.
2. Accessibility/captions gap: plan mentions "accessible modal" but doesn't specify a captions track for the video, nor concrete focus-trap/inertness/scroll-lock mechanics — for a video-centric onboarding feature this is a real WCAG gap, not a nice-to-have. Fix: specify `<track kind="captions">` with a real WebVTT file, plus explicit `role="dialog"`, `aria-modal`, `aria-labelledby`, focus trap + restore, `inert`/`aria-hidden` on background, and scroll lock.
3. Testing scope underspecified: "add focused unit tests" doesn't say what they cover or how — for a stateful modal with multiple dismissal paths or persistence side effects, undertested logic is exactly where regressions hide. Fix: name the actual test file and the specific behaviors under test (gating logic, dismissal paths, focus management).
4. Privacy audit missing: the plan embeds a real screen-recording of the live app — has anyone verified the recording doesn't expose real account balances, transaction descriptions, email addresses, or API keys before committing it to a public repo and shipping it to the web? Fix: review representative frames before committing.
5. Offline-completion-write sequencing ambiguity: the plan says "close optimistically, then save" but doesn't pin down whether the overlay can re-render mid-transition if `profile` updates asynchronously, or whether a slow/offline `saveProfile` could leave the UI in an inconsistent state. Fix: specify the exact ordering — synchronous local state update/close first, async persistence second, with try/catch matching the existing convention.
6. ffmpeg recipe too vague: "re-encode for web" without concrete flags invites an under- or over-compressed result and doesn't mention verification. Fix: give the literal `ffmpeg` command (codec, profile, pixel format, CRF, preset, audio codec/bitrate, `+faststart`) plus an `ffprobe` verification step.
7. No cache strategy for the new binary asset: large static media without explicit cache headers either gets re-fetched on every visit (slow) or risks staleness if ever replaced in place. Fix: use a versioned filename (e.g., `tutorial-v1.mp4`) and add long-lived immutable `Cache-Control` headers via `netlify.toml` `[[headers]]`.
8. Stale-callback / race-condition guard missing: `onAuthStateChanged` is async; if a user signs out and a different user signs in while a `loadProfile`/`saveProfile` call from the previous session is still in flight, the late-resolving callback could write the wrong uid's data into the new session's state. Fix: guard with a ref tracking the "current" uid and bail out if it has changed by the time the async call resolves; also reset `profile`/`tutorialOpen` on sign-out alongside the existing resets.

## Round 3 — Codex
**Remaining findings**
1. Media references inconsistent: PLAN.md still references unversioned poster/caption filenames (`tutorial-poster.jpg`, `tutorial-captions.vtt`) alongside the versioned `tutorial-v1.mp4`, and the `<video>` markup lists `<track>` before `<source>`. Fix: use a monotonically increasing generation token, checked after every async boundary, for any new guarded state — and acknowledge the pre-existing gap explicitly.
4. The privacy-audit record contradicts itself: PLAN.md claims all six frames were inspected, but PLAN-REVIEW-LOG.md (Round 1 response) says only five were; sampled single frames also can't rule out transient PII (a toast, a tooltip) appearing between samples. Fix: author the missing Outro cue and verify all cues against the actual rendered narration audio, correcting any that drift.
6. The accessibility test plan omits focus-trap cycling and cleanup-on-dismissal: the named tests cover initial-focus and focus-restoration but not Tab/Shift+Tab cycling within the dialog, nor that `inert`/`aria-hidden`/scroll-lock are removed/unlocked after every dismissal path (not just the happy path).
