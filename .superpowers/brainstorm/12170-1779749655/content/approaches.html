Three Approaches to Hosting CashFlow

Pick the architecture that fits best — I'll recommend one after you look these over

A

BaaS-First — Firebase Everything ⭐ Recommended

Firebase Auth handles Google login. Firestore stores user data directly from the browser. One Netlify Function handles the AI proxy. No backend server.

Browser → Firebase Auth → Google
Browser → Firestore (user's data)
Browser → Netlify Function → Anthropic API

Pros

  • Least code to write
  • Real-time sync built in
  • Scales automatically
  • Free tier generous (1GB storage, 50k reads/day)

Cons

  • Firestore security rules must be written carefully
  • Firebase vendor lock-in
B

Firebase Auth + Netlify Functions Backend

Firebase Auth for login only. All data reads/writes go through Netlify Functions (your own mini-API). Functions talk to Firestore server-side. More control, more code.

Browser → Firebase Auth → Google
Browser → Netlify Function (CRUD) → Firestore
Browser → Netlify Function (AI) → Anthropic

Pros

  • All business logic server-side
  • No Firestore security rules needed
  • Easier to add rate limiting

Cons

  • Many more functions to write and maintain
  • More latency (extra hop)
  • No real-time sync
C

Offline-First Hybrid (localStorage + Firestore sync)

Keep localStorage as the primary store. Add Firestore as a cloud sync layer — changes push up when online. Works offline, syncs when connected.

Browser → localStorage (immediate)
Background sync → Firestore
Browser → Netlify Function → Anthropic

Pros

  • Works offline
  • Instant UI (no latency)
  • Preserves existing data model

Cons

  • Conflict resolution is hard
  • Most complex to build
  • Cross-device sync can get out of sync
